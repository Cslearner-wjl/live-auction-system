# 需求分析

本文档根据 `ChatGPT-直播竞拍全栈开发.pdf`、课题说明和项目约束整理，用于指导直播竞拍系统的需求边界、业务规则和验收口径。

## 1. 项目目标

构建一套面向抖音直播电商场景的全栈直播竞拍系统，覆盖从商品上架到成交订单的完整链路：

```txt
商品上架 -> 规则配置 -> 直播间展示 -> 实时出价 -> 动态排名 -> 竞拍结束 -> 成交订单
```

系统应重点体现三类能力：

- 业务完整性：主播端、用户端、出价、结算、订单闭环可演示。
- 技术深度：高并发出价一致性、竞拍状态机、WebSocket 房间隔离、断线快照恢复。
- 工程质量：分层清晰、测试覆盖关键规则、文档和 AI 协作记录完整。

## 2. 角色与端侧范围

### 2.1 商家 / 主播端

面向 PC 管理后台，负责商品和竞拍生命周期管理。

核心能力：

- 创建商品，维护商品名称、图片、介绍、卖点标签。
- 配置竞拍规则，包括起拍价、固定加价、竞拍时长、封顶价、自动延时窗口、延时时长、最大延时次数。
- 查看竞拍列表和状态进度。
- 启动竞拍。
- 在竞拍未开始前修改规则。
- 取消异常进行中的竞拍。
- 查看成交、流拍、取消结果。
- 查看订单列表和订单详情。
- 可选使用 AI 生成商品卖点和直播话术。

### 2.2 用户移动端 H5

面向直播间用户，负责观看、出价和接收实时反馈。

核心能力：

- 进入模拟直播间。
- 查看直播间竞拍小卡片。
- 打开竞拍底部半屏面板。
- 查看商品信息、规则摘要、当前价、起拍价、参与人数、倒计时。
- 手动出价，可通过加减按钮按固定幅度调整金额。
- 实时接收当前价、最高出价人、排行榜变化。
- 接收“领先”“被超越”“竞拍延时”“竞拍结束”“成交 / 流拍”等提醒。
- 竞拍结束后查看结果，可选模拟支付。
- 断线重连后拉取最新快照并恢复状态。

## 3. MVP 边界

P0 必须完成：

- 商品创建和竞拍规则配置。
- 起拍价支持 `0`。
- 固定加价幅度校验。
- 封顶价校验和达到封顶价立即成交。
- 主播启动竞拍。
- 用户端进入直播间并查看竞拍卡片。
- 用户端出价。
- WebSocket 同步当前价和竞拍事件。
- 竞拍结束后成交或流拍。
- 成交后生成订单。
- 主播取消异常竞拍。
- 移动端竞拍面板和 PC 后台基础列表。

P1 加分完成：

- 自动延时。
- 排行榜。
- 断线重连快照恢复。
- 并发压测。
- 领先 / 被超越 / 最后 10 秒紧迫感等氛围体验。

P2 可选完成：

- AI 卖点生成。
- 模拟支付。
- 历史竞拍记录。
- 更复杂直播互动和运营标签。

## 4. 关键业务规则

### 4.1 竞拍规则

- 起拍价可以为 `0`。
- 金额全部使用整数分，不使用浮点数。
- 固定加价幅度必须大于 `0`。
- 封顶价必须大于起拍价。
- 竞拍时长必须大于 `0`。
- 延时窗口和延时时长必须为正数，延时时长通常为 10-30 秒。
- 规则只允许在竞拍开始前修改。
- 竞拍中、已结束、已取消的竞拍不得修改核心规则。

### 4.2 状态机

竞拍状态变化必须集中在状态机服务中处理。

推荐状态：

```txt
DRAFT
SCHEDULED
RUNNING
ENDED_SOLD
ENDED_UNSOLD
CANCELLED
```

允许流转：

```txt
DRAFT -> SCHEDULED
SCHEDULED -> RUNNING
RUNNING -> ENDED_SOLD
RUNNING -> ENDED_UNSOLD
RUNNING -> CANCELLED
SCHEDULED -> CANCELLED
```

说明：

- PDF 中提到 `EXTENDED`，但项目实现建议不把延时做成持久状态。
- 延时通过更新 `endTime` 和 `extendedCount` 表达，竞拍仍保持 `RUNNING`。
- 非法状态流转必须抛出业务错误，并写测试覆盖。

### 4.3 出价规则

出价请求格式：

```json
{
  "amountFen": 90000,
  "clientBidId": "uuid-from-client"
}
```

有效出价必须满足：

- 竞拍状态为 `RUNNING`。
- 当前服务器时间未超过 `endTime`。
- 出价人不是当前最高出价人。
- 出价金额大于当前价。
- 出价金额符合固定加价幅度。
- 出价金额不超过封顶价，除非产品明确采用“超过后裁剪到封顶价”策略。
- 同一竞拍下 `clientBidId` 未被处理过。
- 竞拍未结算、未取消。

### 4.4 自动延时

- 当用户在结束前 N 秒内有效出价，系统延长 `endTime`。
- 延时时长通常为 10-30 秒。
- 如果实现最大延时次数，达到上限后不再延时。
- 延时事件通过 WebSocket 广播 `AUCTION_EXTENDED`。

### 4.5 结算规则

- 当前时间到达 `endTime` 时触发结算。
- 有最高出价人则成交，状态变为 `ENDED_SOLD`，生成一个订单。
- 无有效出价则流拍，状态变为 `ENDED_UNSOLD`，不生成订单。
- 出价达到或超过封顶价时立即成交。
- 订单需要保证同一竞拍只生成一次。

## 5. 实时通信需求

### 5.1 房间隔离

事件必须按房间发送，禁止全局广播。

推荐房间：

```txt
room:{roomId}
auction:{auctionId}
user:{userId}
```

### 5.2 核心事件

```txt
AUCTION_STARTED
AUCTION_SNAPSHOT
BID_ACCEPTED
BID_REJECTED
OUTBID
LEADING
AUCTION_EXTENDED
AUCTION_ENDED
ORDER_CREATED
AUCTION_CANCELLED
PING
PONG
```

### 5.3 快照恢复

客户端加载页面或重连时，必须重新拉取快照：

```txt
连接 WebSocket -> 加入房间 -> 获取 AUCTION_SNAPSHOT -> 渲染快照 -> 应用后续事件
```

快照必须包含 `serverTime`，用于前端校准倒计时。

## 6. 后台页面需求

后台至少包含：

```txt
/admin/items
/admin/items/new
/admin/auctions
/admin/orders
```

竞拍列表字段：

- 商品图。
- 商品名。
- 标签。
- 起拍价。
- 固定加价。
- 封顶价。
- 当前出价 / 成交金额。
- 出价次数。
- 竞拍状态。
- 剩余时间。
- 操作按钮。

常用操作：

- 创建竞拍商品。
- 配置竞拍规则。
- 启动竞拍。
- 取消异常竞拍。
- 查看订单。
- 可选生成 AI 卖点。

## 7. 移动端页面需求

核心组件：

```txt
LiveRoomPage
AuctionMiniCard
AuctionPanel
BidStepper
Countdown
BidToast
AuctionResultModal
LiveCommentList
```

交互要求：

- 底部半屏竞拍弹窗。
- 红色主出价按钮。
- 明显倒计时。
- 一键出价。
- 加减按钮按固定幅度调整金额。
- 提交中、已结束、已取消、当前用户已领先时禁用按钮。
- 无人出价展示“起拍价”。
- 有效出价后展示“当前最高价”。
- 成交后展示“落槌价”。
- 当前用户领先展示“当前您已是最高价”。
- 被超越时展示“你已被超越”。
- 最后 10 秒增强紧迫感，但避免重型动画依赖。

## 8. 非功能需求

### 8.1 并发一致性

系统必须保证：

- 当前价不下降。
- 最高出价人唯一。
- 客户端重试不产生重复出价。
- 同一成交竞拍只生成一个订单。
- WebSocket 不发送低于最新价的事件。
- Redis 和数据库可以对账。

推荐方案：

- Redis Lua 脚本原子校验和更新出价热状态。
- 数据库唯一约束保证 `auctionId + clientBidId` 幂等。
- 订单创建走状态机事务。
- 排行榜使用 Redis Sorted Set。

### 8.2 性能目标

基础目标：

- 单直播间 100 在线用户。
- 30 用户短时间内并发出价。

亮点目标：

- 单直播间 1000 WebSocket 连接。
- 100 并发出价。

### 8.3 可观测性

需要结构化记录：

- 竞拍启动、结束、取消。
- 出价接受、拒绝和拒绝原因。
- 延时。
- 订单创建。
- Redis / 数据库对账异常。

日志字段至少包含：

```txt
auctionId
roomId
userId
clientBidId
eventId
```

## 9. 测试与验收

必须覆盖：

- 状态机合法和非法流转。
- `0` 元起拍。
- 非法加价幅度。
- 封顶价校验。
- 开拍后修改规则失败。
- 出价低于或等于当前价失败。
- 出价不符合加价幅度失败。
- 当前最高出价人重复出价失败。
- 竞拍结束后出价失败。
- 重复 `clientBidId` 幂等。
- 封顶价立即成交。
- 成交竞拍只生成一个订单。
- 流拍不生成订单。
- 30 和 100 并发出价。
- WebSocket 房间事件不串房。
- 重连后快照恢复。

## 10. 已知风险

- 出价链路存在 Redis 成功但数据库写入失败的补偿风险，需要事件日志和对账机制。
- 封顶价并发命中时可能重复触发结算，需要订单唯一约束和状态机事务兜底。
- WebSocket 事件可能乱序，客户端应以快照和事件版本为准。
- 倒计时依赖本地时间会漂移，必须使用 `serverTime` 校准。
- AI 功能必须 fallback 到 mock，不能阻塞主流程。
- 课题材料中出现的共享 API Key 不得写入源码、文档示例或前端包。
